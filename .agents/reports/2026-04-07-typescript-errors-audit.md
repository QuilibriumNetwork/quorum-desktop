# TypeScript Errors Audit

> **Warning: AI-Generated**: May contain errors. Verify before use.

## Executive Summary

Running `npx tsc --noEmit --skipLibCheck` initially produced **1,630 errors** (excluding `.native.` platform files). The bulk of these (1,269) were caused by a single bug in `quorum-shared`'s `package.json`: the `exports.import.types` field pointed to `dist/index.d.mts` which was never generated by the build, so TypeScript with `moduleResolution: bundler` couldn't resolve any types from the package.

**After fixing the quorum-shared exports** (PR merged, `v2.1.0-5`), the count dropped to **361 errors**. These remaining errors cluster into 4 well-defined categories.

**Current baseline**: 361 errors (154 production, 207 dev/test).

## Resolved: quorum-shared exports bug (1,269 errors)

The `exports` field in `quorum-shared/package.json` had:

```json
"import": {
  "types": "./dist/index.d.mts",  // never generated by build
  "default": "./dist/index.mjs"
}
```

The build uses `tsc` which only generates `.d.ts`, not `.d.mts`. With `moduleResolution: bundler`, TypeScript followed the `import` condition and found nothing, so all type imports from `@quilibrium/quorum-shared` failed. This caused 262 direct "no exported member" errors (TS2305/TS2724) plus ~1,007 cascading errors from broken prop types (TS2739/TS2740/TS2741/TS2322).

**Fix**: Changed `index.d.mts` to `index.d.ts` in the `import.types` condition. PR merged as `v2.1.0-5`. The primitives migration was complete all along; only the package exports config was wrong.

## Remaining Findings (361 errors)

### Category D: Unused variables and parameters (140 errors)

- **Error codes**: TS6133 (130), TS6196 (8), TS6198 (2)
- **Root cause**: `noUnusedLocals: true` and `noUnusedParameters: true` are enabled in `tsconfig.json`. Declared variables and imported names that are never referenced trigger these.
- **Impact**: **Very low** -- dead code and stale imports. No runtime risk.
- **Note**: ESLint's `@typescript-eslint/no-unused-vars` rule covers much of the same ground (already in warnings). These are the TypeScript compiler's version of the same check.

### Category E: Test environment setup (110 errors)

- **Error codes**: TS2582 (49), TS2304 (61)
- **Root cause**: `src/dev/tests/` files use `describe`, `it`, `expect` (Vitest globals) without the Vitest types being included in the main `tsconfig.json`. The test runner uses a separate config (`vitest.config.ts`), but `tsc` doesn't know about it.
- **Affected files**: Only [src/dev/tests/utils/mentionHighlighting.test.ts](src/dev/tests/utils/mentionHighlighting.test.ts) and [src/dev/tests/utils/mentionUtils.enhanced.test.ts](src/dev/tests/utils/mentionUtils.enhanced.test.ts)
- **Impact**: **Low** -- tests run fine via `yarn test:run` (383 passing). This is purely a `tsc` config gap.
- **Fix**: Add `"types": ["vitest/globals"]` to `tsconfig.json` or exclude test files from the main tsconfig.

### Category C: Implicit `any` parameters (23 errors)

- **Error code**: TS7006
- **Root cause**: Functions where callback/parameter types are not annotated, which `strict: true` flags as implicit `any`. Common pattern: `.map(s => ...)`, `.filter(c => ...)`, inline handlers.
- **Impact**: **Low** -- these are not bugs. TypeScript just cannot infer the type without an explicit annotation. The runtime behavior is correct.
- **Hotspots**: [src/hooks/business/search/useBatchSearchResultsDisplay.ts](src/hooks/business/search/useBatchSearchResultsDisplay.ts) (7 errors), [src/hooks/business/search/useSearchResultsState.ts](src/hooks/business/search/useSearchResultsState.ts) (5 errors), [src/components/context/MessageDB.tsx](src/components/context/MessageDB.tsx)

### Category F: Real type mismatches (88 errors)

- **Error codes**: TS2322 (19), TS4104 (17), TS2345 (16), TS2353 (8), TS2554 (6), TS2352 (4), TS2307 (4), TS18046 (3), TS2774 (2), TS2454 (2), TS7016 (2), TS2741 (1), TS2739 (1), TS2344 (1), TS2339 (1), TS2305 (1)
- **Root cause**: Genuine type errors -- duplicate type definitions diverging from shared types, nullable values passed where required, ref type mismatches, wrong argument types.
- **Impact**: **Medium** -- some represent real potential bugs (e.g. `indexedDbAdapter.ts` has local `UserConfig` diverging from shared `UserConfig`), others are minor strictness issues (e.g. `RefObject<HTMLDivElement | null>` vs `RefObject<HTMLElement>`).
- **Key patterns**:
  - `indexedDbAdapter.ts`: local `UserConfig` type from `src/db/messages` conflicts with `@quilibrium/quorum-shared` `UserConfig`
  - `App.tsx`: potentially-undefined object passed where required
  - `MessageComposer.tsx`, `MessageEditTextarea.tsx`, `SearchBar.tsx`: ref type narrowness (`RefObject<T | null>` vs `RefObject<T>`)
  - `SyncService.ts`: using value as type (`SharedSyncService` vs `typeof SharedSyncService`)
  - `SpaceSettingsModal.tsx`: 6 errors from modal component prop mismatches
- **Hotspots**: [src/dev/tests/services/MessageService.unit.test.tsx](src/dev/tests/services/MessageService.unit.test.tsx) (16), [src/dev/docs/MarkdownViewer.tsx](src/dev/docs/MarkdownViewer.tsx) (22), [src/components/context/MessageDB.tsx](src/components/context/MessageDB.tsx) (11), [src/services/SyncService.ts](src/services/SyncService.ts) (6)
