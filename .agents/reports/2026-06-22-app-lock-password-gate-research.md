# App Lock / Password Gate — Feasibility Research

**Date:** 2026-06-22
**Question:** Can we add a password to block the web app and re-access it? How far can it go — can it genuinely protect local storage at rest, or is it only a UI block? Web vs Electron? Can it sync to mobile? What do other apps do (Telegram, Signal)?

**Scope (locked 2026-06-22):**
- **Primary task:** Option B — a passphrase that **encrypts local storage** and gates the **web/Electron** app at rest. ⚠️ **Pending lead-dev sign-off.**
- **Mobile:** research whether to do the same on `quorum-mobile` (it already has a biometric/phone lock) — **conclusion: yes, it makes sense** (see §6). Same conceptual design, platform-appropriate primitives.
- **Duress protection: go directly with the WIPE option** (Mode B). The **decoy-account** model (Mode A) is documented in §7 for context but is **explicitly out of scope** — too hard to make forensically believable, and a broken decoy is worse than none.
- **Manual wipe: already exists** in the User Settings → Danger Zone on desktop (`DangerZone.tsx`) and on mobile. The duress-wipe should **reuse that existing flow**, triggered by the duress passphrase instead of a typed `RESET` confirmation.

> No prior dedicated research existed. The closest material was the IndexedDB-plaintext finding in [privacy-analysis-quorum-2025-12-03](privacy-analysis-quorum-2025-12-03.md) ("Consider optional at-rest encryption for high-security users") and the key-storage model in [cryptographic-architecture](../docs/cryptographic-architecture.md). This report builds on those.

---

## TL;DR

1. **Direction: Option B — real at-rest encryption** (passphrase → Argon2id → key that encrypts the local DB). Pending lead-dev sign-off. A UI-only lock is security theater for this app's threat model.

2. **A pure UI lock does NOT protect local storage.** Today every sensitive store in `quorum_db` is plaintext IndexedDB. Anyone with file/DevTools access reads it regardless of any lock screen — same posture as Signal Desktop and Telegram Desktop historically.

3. **B is an architecture change, not a toggle.** It touches the whole DB layer, needs a migration for existing users, and forces a lost-passphrase policy decision. Weeks, not days.

4. **Web and Electron differ.** Web: passphrase is the only secret (+ WebAuthn PRF for the master key). Electron: add `safeStorage` (DPAPI/Keychain) — and it *also fixes* a current weakness where Electron stores the master key with **no user secret at all**.

5. **Mobile is worth doing too**, despite the phone lock. The OS lock covers "lost locked phone"; it does **not** cover the journalist threat ("phone unlocked / handed over at a border"). A separate app passphrase + duress mode does. Mechanism: SQLCipher + Argon2id + Keystore-bound key.

6. **Never sync the passphrase.** Per-device local gate everywhere — **Telegram's passcode is explicitly device-local and never synced.** Desktop passphrase + mobile biometric = real parity (same model, platform-appropriate).

7. **Duress = WIPE (decided).** Duress passphrase silently destroys the data. This is what GrapheneOS does, and it's the robust choice. The **decoy-account** alternative is out of scope — very hard to make forensically believable, and a broken decoy is worse than none.

8. **Manual wipe already exists** (`DangerZone.tsx` on desktop + mobile equivalent). Duress-wipe **reuses that flow**, triggered by the passphrase instead of a typed `RESET`. Option B's "destroy the DEK" makes the wipe cryptographically sound (today's reset is local-only and not a secure erase).

## 1. What exists today (verified against current code)

### Key storage
- **Master identity keys** (`UserKeyset`, `DeviceKeyset`) live in a separate `KeyDB` (id=2 on the passkey path, id=1 on the fallback path), two-layer AES-GCM:
  - Outer: an ephemeral, non-exportable AES-GCM key generated by the browser, stored *inside the same IndexedDB record*.
  - Inner: AES-GCM key = `SHA-256(ed448 private key)` on the passkey path.
  - Web with WebAuthn PRF support: the private key itself is wrapped under the passkey's PRF output (`navigator.credentials.get` → PRF → AES key). This is the genuinely strong path.
- **Everything else** in the main `quorum_db` is **plaintext**: `messages`, `conversations`, `encryption_states` (ratchet state = symmetric session keys!), `space_keys` (per-space private keys as hex), `user_config`, `space_members`, `action_queue`, `search_indices`.

### Electron reality (important)
`usePasskeyFlow` detects `window.electron` and **immediately sets `quorum-master-prf-incompatibility=true`**, forcing the **fallback path**. On Electron the master private key is stored in `KeyDB id=1` protected *only* by the ephemeral in-DB AES key + OS filesystem permissions — **no user secret, no passkey**. Electron does **not** currently use `safeStorage`, `keytar`, or any OS keychain.

### App-lock today
**None.** There is no lock screen, no idle timer, no re-auth on resume, no PIN/password entry. `App.tsx` reads `completedOnboarding` from localStorage on mount and logs the user straight in. Closing and reopening requires no gesture.

### Mobile (quorum-mobile, separate repo) for contrast
Private keys stored individually in `expo-secure-store` → iOS Keychain `.whenUnlockedThisDeviceOnly` / Android Keystore. There is a **biometric gate** (`expo-local-authentication`, Face ID / Touch ID / fingerprint, passcode fallback). So mobile *already* has the strong, OS-backed equivalent of an app lock — desktop/web has nothing.

## 2. The two designs — how far each goes

### Option A — UI Lock (cosmetic gate)
A lock screen overlay; password (or its hash) stored locally; on unlock the app proceeds. Optionally an idle/auto-lock timer and a lock-on-resume handler.

| Property | Result |
|---|---|
| Stops a casual onlooker / shoulder-surfer | ✅ Yes |
| Stops someone with the unlocked machine from opening the app | ✅ Yes |
| Protects IndexedDB on disk | ❌ **No** — all stores stay plaintext |
| Stops DevTools / file-system / forensic access | ❌ **No** |
| Web | ✅ Works |
| Electron | ✅ Works |
| Effort | **~1-2 days** |

The password here is a **bouncer at the door of a glass house**. The honest reading is: it deters a person, it does not protect the data. We would have to say that plainly in the UI, or it's security theater.

### Option B — At-Rest Encryption (real protection)
Derive a key from the password with a slow KDF (**Argon2id** preferred, PBKDF2 as floor) and encrypt the sensitive IndexedDB payloads with it. The app is unusable until the password decrypts the data into memory.

| Property | Result |
|---|---|
| Protects IndexedDB on disk | ✅ Yes — ciphertext without the password |
| Stops DevTools-at-rest / file-system / forensic access | ✅ Yes (while locked) |
| Stops a running, unlocked session being inspected | ❌ No (keys are in memory once unlocked — unavoidable, same as Signal/everyone) |
| Web | ✅ Works (Web Crypto + WASM Argon2) |
| Electron | ✅ Works, and can be hardened with `safeStorage` |
| Effort | **Weeks** — touches the whole DB layer; migration of existing users; lockout/recovery design |

Key cost drivers beyond crypto:
- **Migration**: existing users have plaintext DBs to re-encrypt on first set-up.
- **Lost-password = lost-data** (or a backup-key escape hatch). Must be decided and communicated.
- **Performance**: encrypt/decrypt on every read/write, or hold a decrypted working set in memory.
- **The master keys are the higher-value target** and they're already (mostly) protected — the *messages and space keys* are what this would newly protect.

### Where the password could plug in (cleanest approach)
Rather than encrypting every store record-by-record, the high-leverage move is to make the **password an additional factor that protects the key material**, and/or wrap the `encryption_states` + `space_keys` + message blobs under a password-derived key. On Electron, store the password-derived key (or a random data-encryption key) inside `safeStorage` so the OS user binding does the heavy lifting and the password adds the second factor.

## 3. Web vs Electron — they are genuinely different

| | Web (browser) | Electron (desktop) |
|---|---|---|
| Available secret stores | Password only; WebAuthn PRF for the master key | Password + **`safeStorage`** (DPAPI on Win, Keychain on macOS, libsecret on Linux) |
| Passkey/PRF path | Used when browser supports it (strong) | **Disabled** — forced to fallback (weak today) |
| OS-user binding | None | Yes via `safeStorage` (protects vs *other users*, **not** vs malware in same session) |
| Best realistic posture | Password-derived KDF + Web Crypto | `safeStorage`-wrapped DEK + optional password second factor |

`safeStorage` is the single biggest *desktop-only* win and would also fix the current Electron weakness (master key with no user secret). It protects against other users on the machine and other apps; it does **not** protect against malware running as the same user (that's a documented, universal limitation — same for Signal/VSCode/everyone).

## 4. Does it sync to mobile? (the Telegram question)

**No — and that is the correct, industry-standard answer.**

- **Telegram's passcode is explicitly device-local and never synced or tied to the account.** You set it separately on each device; forgetting it on mobile means reinstalling. It is a local lock, full stop.
- **Signal** has no cross-device app password either; at-rest is an OS/keychain concern per device.
- The reason is structural: an app-lock password protects *local* data on *one* device. Syncing it would mean transmitting the password (or a derived secret) over the network — which defeats the point and creates a new attack surface. The thing you'd sync isn't security, it's a preference.

For Quorum specifically:
- Mobile **already** has a stronger gate (biometrics + OS keychain). It doesn't need our app-password.
- A desktop/web password and a mobile biometric lock can coexist as **independent local gates** — which is exactly what Telegram does. "Parity" here means *each platform has a lock appropriate to its environment*, not *the same secret on both*.
- ⚠️ Per the QNS/lead-dev pattern, anything claiming "syncs across devices" must go through the lead dev. But the recommendation here is the no-sync design, so there's likely nothing to clear — it's the default behavior everywhere.

## 5. What other apps do — summary

| App | App-lock? | At-rest DB encryption | Synced across devices? |
|---|---|---|---|
| **Telegram** | Yes (passcode) | Cloud chats not E2E; local cache plaintext-ish | **No — device-local by design** |
| **Signal Desktop** | No app password historically (feature-requested since 2017) | SQLCipher DB; key was *plaintext in config.json* until 2024, now `safeStorage` | No |
| **Element/Matrix** | Varies; web has no robust at-rest | IndexedDB; keys via passphrase-backed secure backup | No (passphrase is for key backup, not an app lock) |
| **Quorum (today)** | **No** | Master key encrypted (web/PRF) / weak (Electron); everything else plaintext | n/a |
| **Quorum mobile** | **Yes (biometric)** | OS Keychain/Keystore | No |

The pattern is unambiguous: **app locks are local, not synced.** The serious players moved toward **OS-keychain-wrapped DB keys** (Signal's 2024 `safeStorage` migration is the reference example).

## 6. Mobile (quorum-mobile) — does Option B add value on top of the phone lock?

**Conclusion: yes, do it — lower urgency than desktop/web, different mechanism.** The phone lock covers the "lost locked phone" case; it does **not** cover the journalist case ("phone unlocked / handed over"), which is exactly where a separate app passphrase + duress-wipe earns its keep.

### What mobile has today
- Private keys in `expo-secure-store` → iOS Keychain (`.whenUnlockedThisDeviceOnly`) / Android Keystore, gated by biometrics (`expo-local-authentication`). The OS already encrypts app storage at rest, tied to device unlock.
- So "someone picks up your locked phone and dumps the filesystem" is **already** largely covered by the OS — much better than the desktop/web starting point.

### Why an *additional* in-app passphrase still matters on mobile
The phone lock and the app passphrase defend against **different adversaries**:

| Adversary | Phone lock covers it? | App passphrase adds? |
|---|---|---|
| Lost/stolen locked phone, opportunist | ✅ Yes | marginal |
| **Phone handed over unlocked** (border, checkpoint, coercion) | ❌ No — it's already unlocked | ✅ Yes — second gate |
| **Forensic extraction tools** (Cellebrite/GrayKey-class) | ⚠️ Partial; depends on OS/exploit | ✅ Yes — app data is under a *separate* key not released by OS unlock |
| Malware as the user | ❌ No | ⚠️ Only while locked |

The journalist scenario is precisely "phone is unlocked / I'm forced to unlock it." That's where the OS lock gives nothing and a separate app passphrase (and especially a **duress** variant, §7) is the whole point.

### Mobile mechanism for Option B
- Encrypt the app DB with **SQLCipher** (AES-256, encrypts tables, indexes, journals — the mature standard; Guardian Project / OWASP MASTG recommend it). Web/desktop would do the equivalent over IndexedDB payloads.
- Key = **Argon2id(passphrase)**, *optionally* combined with a Keystore/Keychain-held secret so brute-force must happen on-device (hardware-bound) rather than offline against an extracted file. OWASP MASTG explicitly endorses "biometric data combined with keystore" + Argon2/PBKDF2 for the SQLCipher key.
- Biometrics stay as the *convenience* unlock for the **real** account; the passphrase is the root of trust and the duress entry point (you can't reasonably hide a decoy behind a fingerprint).

### Cross-platform design note
Same **conceptual** design across web/Electron/mobile (passphrase → KDF → DEK → encrypted store), different **primitives** (IndexedDB+WebCrypto / IndexedDB+safeStorage / SQLCipher+Keystore). The passphrase is **per-device and never synced** (§4). This is real parity: same security model, platform-appropriate implementation.

## 7. Duress / Plausible Deniability — the journalist feature

> **DECISION (2026-06-22): duress = WIPE (Mode B).** The decoy-account model (Mode A) is **out of scope** — kept below only to document *why* we rejected it. This section's job now is (1) lock the wipe design and (2) record the decoy reasoning so we don't relitigate it later.

There are three possible "second password" behaviors. We chose the middle one:

| Mode | What the duress password does | Reversible? | Forensic difficulty | Status |
|---|---|---|---|---|
| **A. Decoy account** | Opens a *believable, functional* dummy account; real data stays hidden | Real data preserved | **Very hard to do right** | ❌ **Out of scope** |
| **B. Duress wipe** | Silently destroys the real keys/data, drops to clean first-run | **No — destroys data** | **Easiest to make robust** | ✅ **Chosen** |
| **C. Plain second lock** | Just a normal app lock (no deniability) | n/a | n/a | ➖ Doesn't serve the goal |

### Mode B — Duress Wipe (CHOSEN)
Entering the duress password **destroys the decryption key** (and ideally overwrites the encrypted DB), rendering all data permanently unrecoverable, then optionally drops you into a clean first-run state.

- **Why it's strong:** there is no hidden data to *find* after the wipe — the deniability problem (below) largely evaporates. GrapheneOS chose exactly this: "redirecting to a secondary profile is not as secure as triggering a full device reset." The wipe is silent — no confirmation, no announcement.
- **Why it's tractable for us:** we don't manage block-level storage, but we *do* control the DEK. "Forget the key + clear the encrypted stores" is a deletion we can implement cleanly on all three platforms.
- **Caveats:** (1) genuinely irreversible — a mistyped duress password = data loss, needs very careful UX; (2) network side-effects — if a wipe also tombstones the account/keys on the Quilibrium side, that's a separate, irreversible protocol action to design; (3) a *local-only* wipe doesn't remove what's already on the network/other devices.

### Mode A — Decoy Account (REJECTED — reasoning recorded so we don't relitigate)
Two passphrases derive two independent encrypted volumes; each opens a fully usable account, neither reveals the other.

- **Cryptographically straightforward:** two KDF salts → two DEKs → two encrypted partitions. Each partition is just ciphertext; without the matching password it's indistinguishable from random.
- **The hard part is NOT the crypto — it's deniability under forensics.** Research is blunt here: hidden volumes "can be detected and their size estimated using multiple copies of encrypted containers" (shadow copies / disk imaging over time), via **non-random free space, entropy anomalies, timestamps, and metadata**. VeraCrypt itself had an advisory where a hidden volume left *deterministic plaintext markers / zeroed sectors* where random ciphertext was expected. Translated to our app:
  - **The decoy must look lived-in.** An empty decoy is an obvious decoy. It needs plausible history, contacts, timestamps — which is a *product* problem, not a crypto one.
  - **Storage-size accounting must not leak the hidden volume.** If the decoy account is 5 MB but the IndexedDB/SQLCipher file is 500 MB, the adversary infers hidden data.
  - **No app-level metadata may betray two accounts** — a single `passkeys-list` entry, one config record, one set of timestamps, one device registration on the network. A sophisticated adversary with the source code (Quorum is open-source!) knows exactly what artifacts to look for.
  - **Network observability:** if the real account is active on the Quilibrium network, traffic/registration tied to the device can contradict "this is my only account." Network-layer deniability is a much deeper problem than local deniability.
- **Honest verdict:** a decoy that fools a *casual* inspector (a border agent scrolling the phone) is **very achievable**. A decoy that fools a *resourced forensic adversary* (the actual journalist threat) is **extremely hard and easy to get subtly wrong** — and a broken deniability feature is *worse than none*, because it gives users false confidence in exactly the situation where being caught is dangerous. This is the recurring warning across the literature: duress/decoy features "won't meet their intended objective — there will always be traces."

### Recommendation within duress
1. **Ship Mode B (duress-wipe) first** if we do duress at all — it's the most robust, hardest to defeat, and aligns with GrapheneOS's reasoning. Pair with careful irreversibility UX and a decision on whether it also wipes network/other-device state.
2. **Treat Mode A (decoy) as a research-grade R&D track**, not a near-term ship. If pursued, scope it explicitly as "deters casual/coercive inspection" and be honest in the UI about what it does *not* defend against (forensic imaging). Do **not** market it as forensic-proof.
3. Either way this needs a **threat-model document** naming the adversary tiers (shoulder-surfer → border agent → forensic lab → nation-state) and stating, per tier, what each mode does and doesn't promise. For a privacy-first app serving journalists, overpromising here is a safety issue, not just a marketing one.

## 6b. Wipe — what exists today and what duress-wipe reuses

**A manual wipe already ships** (verified in code, 2026-06-22):
- Desktop: `src/components/modals/UserSettingsModal/DangerZone.tsx` → "Reset App Data". Requires typing `RESET`, then: `queryClient.clear()` → `indexedDB.deleteDatabase('quorum_db')` → `localStorage.clear()` → `sessionStorage.clear()` → hard reload.
- Mobile: equivalent reset exists in the mobile User Settings (per user; confirm exact file when implementing).

**Two gaps in the current reset that matter for a *security* wipe:**
1. **Local-only.** It deletes the browser DB but does **not** touch the account/keys on the Quilibrium network or other devices. Fine for "clear this browser," but a duress wipe should make an explicit decision about network/other-device state (lead-dev question).
2. **Not a secure erase.** `deleteDatabase` removes the records but does not cryptographically guarantee the bytes are unrecoverable. **Option B makes this moot:** if the data is encrypted at rest, "destroy the DEK" instantly renders the ciphertext useless even if the raw bytes linger. This is *why* duress-wipe should sit on top of Option B, not on plaintext storage.

**Duress-wipe = the existing reset flow, triggered by the duress passphrase** (no `RESET` prompt, silent), with the DEK-destruction guarantee from Option B layered in.

### Wipe triggers as a category (all reuse one primitive)

| Trigger | In scope? | Notes |
|---|---|---|
| **Manual "Reset App Data"** | ✅ Exists | Keep; optionally harden to DEK-destroy once Option B lands |
| **Duress-passphrase wipe** | ✅ **v1** | Reuses the reset flow; silent; the journalist feature |
| **Failed-attempts wipe** | ◐ Optional/opt-in | Anti-brute-force; accidental-self-wipe risk |
| **Remote wipe** | ⏸ Out of scope | Needs authenticated network channel |
| **Dead-man's / timeout wipe** | ⏸ Out of scope | Niche; high accidental-loss risk |

Core primitive shared by all: **destroy the DEK + clear stores.** Build it once (it's part of Option B's foundation), layer triggers on top.

## 6. Recommendation

**Direction confirmed: Option B (real at-rest encryption), pending lead-dev sign-off.** The rest is sequencing.

### Build order (each layer reuses the one below it)
1. **Foundation — encrypted stores + key management.** Passphrase → **Argon2id** → DEK; encrypt `space_keys`, `encryption_states`, message content, `user_config`. Web/Electron over IndexedDB (WebCrypto); mobile over **SQLCipher** + Keystore/Keychain. On Electron add `safeStorage`; on mobile bind the DEK with the Keystore so brute-force is on-device only. Decide the **lost-passphrase policy** (lost password = lost data, or an escape-hatch backup key) up front.
2. **Lock UX** — lock screen, auto-lock-on-idle, lock-on-resume. Cheap once (1) exists.
3. **Wipe primitive** — "destroy DEK + clear encrypted stores." Builds on the **existing `DangerZone` reset flow** (§6b); shared engine for manual wipe and duress-wipe.
4. **Duress-wipe** — **the chosen duress mode.** Duress passphrase silently triggers the wipe (no `RESET` prompt). Thin layer on (3). Lead-dev decision: does it also wipe network/other-device state.

**Out of scope:** decoy account (Mode A, §7) — documented for context only. Too hard to make forensically believable; a broken decoy is a user-safety regression, not a feature.

### Hard rules carried from the research
- **Do not sync the passphrase.** Per-device local gate on every platform — the Telegram model. Mobile biometric lock and desktop passphrase coexist as independent gates; that *is* parity.
- **A UI-only lock is security theater** for this app's threat model. We're past that — B is the call.
- **Mobile still benefits** despite the phone lock, because the journalist threat is "phone is already unlocked / handed over," which the OS lock doesn't cover (§6).
- **Never overpromise deniability.** A broken decoy is worse than none. Anything shipped must state its adversary tier honestly — this is a user-safety obligation, not marketing.

### Next steps
- **Lead-dev call** (ask, per the QNS/Telegram-not-GitHub pattern): green-light Option B; and specifically the two irreversible/network-touching decisions — (a) does duress-wipe also tombstone the account on the Quilibrium network, and (b) is there any account-level deniability obligation we can't meet because of network-side registration?
- **Threat-model doc** naming adversary tiers (shoulder-surfer → border agent → forensic lab → nation-state) and what each mode promises per tier. Prereq for honest UI copy.
- **Brainstorming/spec session** to lock the lost-passphrase policy and the build-order cut line (almost certainly: ship 1-4, defer 5).
